# Legal Production RAG 

A production-ready **FastAPI** chat API wrapping a **LangGraph agentic-RAG** agent that
answers questions about the **Constitution of India** and returns **verifiable citations**
(Article + page) with every answer.

Instead of a single retrieve-then-answer pass, the agent runs a self-correcting
`retrieve → grade → rewrite → generate` loop: it grades retrieved passages for relevance,
reformulates the query and retries when nothing useful comes back, and answers strictly
from the surviving excerpts. Citations are read from the source document's metadata — never
generated by the model — so they can't be hallucinated.

## About the corpus: the Constitution of India

The agent answers over the **Constitution of India** — the longest written constitution of any
sovereign country in the world.

- **~145,000 words** in the English version (≈ 146,385 words), spread across roughly
  **117,000 words of the original text** plus amendments.
- **Originally 395 articles** in **22 parts** and **8 schedules**; through amendments it now
  has roughly **448 articles**, **25 parts**, and **12 schedules**.
- **Over 100 amendments** have been made since it came into force.
- It draws from many sources — e.g. Fundamental Rights (USA), parliamentary system (UK),
  Directive Principles (Ireland), and the federal structure with a strong centre (Canada).

> These scale figures are also why **hybrid retrieval** matters here: exact "Article N" /
> "Schedule N" lookups (BM25) and conceptual queries (dense vectors) cover very different
> question styles over a long, densely cross-referenced document.

## Features

- **Agentic RAG self-correction loop** — grade retrieved passages, rewrite the query and
  retry on a miss, then generate an answer grounded only in the surviving excerpts.
- **Hybrid retrieval** — an ensemble of BM25 (exact "Article N" / "Schedule N" lookups) and
  dense vector search (conceptual queries), weighted together.
- **Verifiable, non-hallucinated sources** — every answer carries `{article, page, part}`
  citations derived from chunk metadata.
- **Security pipeline** — prompt-injection detection, delimiter cleaning, PII masking on
  input, and PII/harmful-content checks on output.
- **Response caching** — in-memory TTL cache keyed on the cleaned message.
- **Rate limiting** — per-IP limits via `slowapi`.
- **Observability** — structured JSON logging, in-memory metrics, and optional LangSmith
  tracing.
- **Operational endpoints** — health, metrics, and cache-stats endpoints for
  Docker/Kubernetes.
- **Containerized** — ships with a `Dockerfile` and `docker-compose.yml`.

## Architecture

### Request pipeline

Every `POST /chat` request flows through a fixed pipeline in `app/main.py`:

1. **Security input check** — prompt-injection block, delimiter cleaning, PII masking.
   Blocked input returns `400`.
2. **Cache lookup** — a hit returns immediately (`model_used="cache"`), citations included.
3. **Agent invoke** — the LangGraph agentic-RAG graph.
4. **Security output check** — PII masking + harmful-content blocking on the answer text
   (structured citations bypass masking).
5. **Cache store** — persists the full `{response, sources, model_used}`.
6. **Metrics + structured log**, then return the response.

### The agent graph

The agent (`app/agent.py`) is a LangGraph `StateGraph` over a `RAGState`:

```
            ┌──────────┐
   START ──▶│ retrieve │◀─────────────┐
            └────┬─────┘              │
                 ▼                    │
            ┌──────────┐         ┌─────────┐
            │  grade   │         │ rewrite │
            └────┬─────┘         └────▲────┘
                 │                    │
        route_after_grade            │
       ┌─────────┼──────────┐        │
       ▼         ▼          ▼        │
 ≥1 survivor  0 + retries  0 + no    │
       │      left ────────┼─retries │
       ▼                   │  left   │
 ┌──────────┐              └─────────┘
 │ generate │         ┌──────────┐
 └────┬─────┘         │ fallback │
      ▼               └────┬─────┘
     END ◀─────────────────┘
```

- **retrieve** — query the hybrid retriever (using the rewritten query if one exists).
- **grade** — one batched call labels each passage relevant / not. **Fail-open**: an
  unparseable reply keeps all passages so `generate` is never starved.
- **rewrite** — reformulate the query and retry (bounded by `MAX_RAG_RETRIES`, default 1).
- **generate** — answer strictly from the surviving excerpts. `_make_sources()` derives the
  deduped, sorted citations from those excerpts' metadata.
- **fallback** — a graceful "couldn't find relevant provisions" message when retries are
  exhausted with no relevant passages.

Routing (`route_after_grade`) is purely on survivor count: `≥1` → generate; `0` with retries
left → rewrite → retrieve; `0` out of retries → fallback.

### Models

- **Primary — Gemini** (`gemini-3.1-flash-lite`) runs every reasoning step: grade, rewrite,
  generate.
- **Fallback — Ollama** (local) is used *only* to write the final answer if Gemini errors at
  generate time. It is not on the normal hot path. If both fail, a static apology is
  returned (`model_used="error_handler"`).

## Retrieval & ingestion

The corpus is static, so it is embedded once, offline.

- **Ingestion** (`app/ingest.py`, a one-time CLI) parses the PDF with a line-leading article
  regex and Part/Schedule detection, carrying the active Article/Part across page breaks. It
  tags each chunk with `{article, part, page}` metadata, embeds the chunks, and persists a
  Chroma collection plus `chroma_db/chunks.jsonl`. Amendment footnotes and running headers are
  deliberately kept as minor noise rather than stripped (see
  [ADR 0003](docs/adr/0003-keep-amendment-footnotes.md)).
- **Embeddings** — local `qwen3-embedding:4b` (2560-dim) served by Ollama, used symmetrically
  at ingest and query time. No API key, no quota.
- **Hybrid retriever** (`app/retriever.py`) — rebuilds BM25 from `chunks.jsonl` at startup
  and loads the persisted Chroma vector store, combining them in an `EnsembleRetriever`
  (default weights 0.4 BM25 / 0.6 vector). The API loads the persisted store; it does not
  embed the corpus at request time.

## Setup & run

```bash
uv sync                                 # install dependencies
ollama pull qwen3-embedding:4b          # one-time: pull the embedding model (Ollama must be running)
cp .env.example .env                    # then fill in GOOGLE_API_KEY (and LangSmith keys if used)

uv run python -m app.ingest             # one-time: build ./chroma_db from the PDF
uv run python -m app.ingest --dry-run   # optional: parse + report chunks only, no embedding

uv run uvicorn app.main:app --reload    # dev server on http://localhost:8000
```

The ingest step must be run once before the API starts — it builds the persisted vector
store the API loads at startup.

## Configuration

Settings are loaded from environment variables / `.env` (see `.env.example`). Only
`GOOGLE_API_KEY` is required; everything else has a sensible default.

| Variable               | Default              | Description                                             |
| ---------------------- | -------------------- | ------------------------------------------------------ |
| `GOOGLE_API_KEY`       | _(required)_         | API key for the primary Gemini model                   |
| `LANGCHAIN_TRACING_V2` | `true`               | Enable LangSmith tracing                                |
| `LANGCHAIN_API_KEY`    | _(empty)_            | LangSmith API key                                      |
| `LANGCHAIN_PROJECT`    | `production-api`     | LangSmith project name                                 |
| `APP_ENV`              | `development`        | Environment (`development` / `production`)             |
| `LOG_LEVEL`            | `INFO`               | Logging level                                          |
| `RATE_LIMIT`           | `20/minute`          | Per-IP rate limit on `/chat`                           |
| `CACHE_TTL_SECONDS`    | `3000`               | Response cache time-to-live                            |
| `MAX_RETRIES`          | `3`                  | Generic retry budget                                   |
| `EMBEDDING_MODEL`      | `qwen3-embedding:4b` | Ollama embedding model (changing it requires re-ingest) |
| `CHROMA_PERSIST_DIR`   | `./chroma_db`        | Persisted vector-store directory                       |
| `MAX_RAG_RETRIES`      | `1`                  | Query rewrites before giving up                        |

Additional retrieval defaults (`chunk_size`, `chunk_overlap`, `retrieval_k`, `bm25_weight`,
`vector_weight`, `body_start_page`, …) live in `app/config.py`.

```bash
curl -s http://localhost:8000/chat \
  -H 'Content-Type: application/json' \
```

```json
{
  "response": "Article 21 guarantees that no person shall be deprived of life or personal liberty except according to procedure established by law.",
  "thread_id": "default",
  "model_used": "primary",
  "cached": false,
  "processing_time_ms": 842.13,
```

`article` and `part` are `null` for the Preamble and Schedules, which are cited by page only.

## Docker

```bash
uv run python -m app.ingest             # build ./chroma_db on the host first
docker compose up --build
```

The vector store is built on the host and mounted into the container read-only
(`./chroma_db:/app/chroma_db:ro`). Because the vector leg embeds each query, the container
needs a reachable Ollama instance — e.g. set `OLLAMA_HOST=http://host.docker.internal:11434`.

## Testing

```bash
uv run pytest                           # run the full suite
uv run pytest tests/test_security.py    # run one file
```

Tests cover the cache, security, and ingest-parser layers. They make **no LLM calls and no
embeddings**, so they are fast, free, and deterministic. The agent, retriever, and full
request pipeline are exercised manually (they require the built index and live models).
